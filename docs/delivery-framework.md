# 交付框架

> 基于 Forge 交付方法论 v2.0 裁剪，适配抛弃型 PoC 场景。
> 核心理念不变：**用文档对抗 AI 遗忘，用验收测试对抗质量腐化，用知识沉淀对抗经验流失。**

---

## 一、交付模型

### 1.1 两层迭代结构（裁剪自三层）

PoC 项目规模较小，去掉 Turn 级底线自动检查，保留 Phase 和 Session 两层：

```
Phase（里程碑）
├─ 粒度：1 个可验收的功能集
├─ 周期：1-4 个 Session
├─ 产出：可运行的系统 + 验收结论
└─ 闭环：验收 → 经验编码

Session（最小工作单元）
├─ 粒度：1 次 Claude Code 对话（~2-3 小时）
├─ 周期：固定 PDCA 微循环
├─ 产出：代码 + 测试 + WORK_LOG 条目 + Git commit
└─ 闭环：验证 → 记录 → 提交
```

### 1.2 四维项目记忆体系

AI 协同交付的核心约束是**上下文的连续性**。用四个维度构建项目记忆，对抗 AI 遗忘：

| 维度 | 载体 | 回答的问题 |
|------|------|-----------|
| **时间** | `WORK_LOG.md` | 发生了什么？为什么这样做？ |
| **空间** | 代码 + 架构文档 | 当前状态是什么？ |
| **质量** | 验收测试 + Bug 记录 | 什么能用？什么不能用？ |
| **知识** | `CLAUDE.md` | 我们学到了什么？ |

> Forge 经验：Session 10 之前知识全靠人脑记忆，导致同类问题反复出现。建立四维记忆后，Bug 复发率降为零。

### 1.3 Session PDCA 循环

每个 Session 遵循固定结构，**不允许跳步**：

```
Plan  → 声明本 Session 目标（1 句话），记录到 WORK_LOG
Do    → 实施（编码 + 测试）
Check → 运行验证 + 变更清单确认 + 统计快照
Act   → 经验沉淀 + Git commit + WORK_LOG 更新
```

**统计快照**：每个 Session 结束时在 WORK_LOG 中记录关键数值（文件数、代码行数、测试数、Bug 数等），提供客观进度度量。

### 1.3 Phase 规划

| Phase | 目标 | 验证重点 |
|-------|------|---------|
| 0 | 项目初始化 + 交付框架建立 | 工程就绪 |
| 1 | 核心业务建模 + 基础架构 | 技术可行性 |
| 2 | AI 原生交互体验 | **用户交互体验** |
| 3 | 智能决策辅助 | **AI 智能体验** |
| 4 | 安全加固 + 权限体系 | **安全性** |
| 5 | 集成验证 + 结论输出 | 整体验证 + 交付结论 |

> Phase 编号一旦确定，永不更改。需要插入中间阶段用小数点扩展（如 Phase 2.5）。

## 二、质量保障

### 2.1 三道防线（裁剪自四道）

```
第 1 道：本地快速验证（30 秒反馈）
  └─ 编译检查 + 单元测试 + lint

第 2 道：验收测试（Phase 级）
  └─ 场景先行，编码前写预期，通过率 ≥ 90%

第 3 道：安全检查（Phase 级）
  └─ 无硬编码密钥、输入校验、XSS/注入防护
```

### 2.2 验收测试驱动（ATDD）

```
Phase 编码前          Phase 编码后          Phase 验收
    │                    │                    │
    ▼                    ▼                    ▼
 写场景标题           交叉验证校准         运行 + 回填数据
 + 关键预期           + 数据校正           + 统计通过率
```

关键原则：
- **场景先行**：编码前写验收场景标题 + 预期结果
- **安全不裁剪**：安全相关验收测试覆盖率与生产标准一致
- **性能可裁剪**：大规模调用场景记录设计预留，不做压测

### 2.3 Bug 治理

**闭环管理**：每个 Bug 必须包含编号、根因分析、修复方案、模式识别。

```
BUG-001: [标题]
- 现象：...
- 根因：[枚举序列化 | 多入口遗漏 | 空值处理 | UI 交互 | ...]
- 修复：...
- 全局排查：是否存在同类问题？已排查 / 已修复 N 处
```

关键纪律：
- 发现一个 Bug 属于系统性问题时，**立即全局排查**同类问题并一次性修复
- Bug 记录在 WORK_LOG 中，标注根因类型
- 修复率目标 ≥ 95%

> Forge 经验：枚举序列化 Bug 出现 3 次后才做全局排查。建立纪律后一次 `grep -r "enum"` 找到 3 处同类问题，一次性修复，零复发。

## 三、知识管理

### 3.1 经验编码管道

```
发现经验 → WORK_LOG 即时记录 → 跨 Session 验证 → 编码到 CLAUDE.md
```

PoC 项目不建立 Skill / Baseline 层，所有经验直接编码到 CLAUDE.md。

### 3.2 双基线比对机制

```
plan-baseline.md                    design-baseline.md
"我们想要构建什么"                    "我们实际构建了什么"
     │                                    │
     │        每 Phase 交付后比对           │
     └──────────────┬─────────────────────┘
                    │
                    ▼
         comparison-phaseN.md
         ✅ 已实现  ✂️ 已裁剪  ⏳ 待实现  🔄 变更
```

- **plan-baseline**：人 + AI 共创，定义功能规划、优先级、验收标准
- **design-baseline**：每 Phase 交付后从代码实现反写（AI 扫描 + 人工确认）
- **比对报告**：逐条比对，标注状态和差异原因，指导后续 Phase 调整

### 3.3 文档体系

| 文档 | 用途 | 更新频率 |
|------|------|---------|
| `README.md` | 项目定位与验证重点 | 按需 |
| `WORK_LOG.md` | 每日工作记录（替代 logbook） | 每 Session |
| `CLAUDE.md` | AI 协作指令与沉淀经验 | 经验验证后 |
| `docs/delivery-framework.md` | 交付框架（本文档） | 按需调整 |
| `docs/baselines/plan-baseline.md` | 规划基线 | 共创 + Phase 调整 |
| `docs/baselines/design-baseline.md` | 设计基线（代码反写） | 每 Phase 交付后 |
| `docs/baselines/comparison-phaseN.md` | 基线比对报告 | 每 Phase 交付后 |
| `docs/acceptance/` | 验收测试用例 | 每 Phase 开始 |

### 3.4 文档债务控制

- WORK_LOG 每 Session 必须更新，不更新不提交
- 每 3 个 Session 做一次文档回顾，清理过时内容

## 四、人 + AI 协作模式

### 4.1 角色分工

| 维度 | 人类（决策者） | AI（执行者） |
|------|--------------|------------|
| 方向决策 | 定义 Phase 目标和优先级 | 评估可行性、提供方案 |
| 架构设计 | 最终拍板 | 分析利弊、推荐方案 |
| 代码实现 | 审查、验收 | 生成、修复、重构 |
| 质量守护 | 定义验收标准 | 执行测试、修复 Bug |
| 经验沉淀 | 确认有效性 | 记录 WORK_LOG、编码到 CLAUDE.md |

### 4.2 协作原则

- **声明式意图**：人类说"做什么"，AI 自主执行并交付结果
- **并行 Agent**：独立任务使用 2-3 个并行 Agent 加速
- **粒度提交**：每个逻辑变更一个 commit，可 bisect、可回滚
- **本地验证优先**：先本地验证通过，再做集成验证

### 4.3 反模式清单

| 反模式 | 代价（Forge 实证） | 纠正方式 |
|--------|-------------------|---------|
| 后补验收测试 | Session 13 发现 16 处错误 | 场景先行，编码前写预期 |
| 大杂烩提交 | 难以 bisect 和回滚 | 每个逻辑变更一个 commit |
| 跳过 PDCA 中的 Check | 同类问题跨 Session 重复出现 | 必须运行验证 + 统计快照后再提交 |
| 经验不编码 | Session 5 的 Docker 坑在 Session 14 再踩 | 验证有效后 5 分钟内写入 CLAUDE.md |
| 单入口测试 | Session 14 REST 通但 WebSocket 不通 | 同一功能多入口测试 |
| 过度自动化 | 用户反馈"黑盒感" | 关键决策点人工审批 + 执行透明度 |

## 五、度量指标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 验收测试通过率 | ≥ 90% | 低于 90% 说明质量防线有漏洞 |
| Bug 修复率 | ≥ 95% | 低于 95% 说明有遗留问题积累 |
| Bug 复发率 | 0% | >0% 说明全局排查纪律未执行 |
| 安全检查通过率 | 100% | 安全不打折 |

## 六、PoC 特别约定

### 裁剪项（与生产系统的差异）

| 维度 | PoC 做法 | 设计预留 |
|------|---------|---------|
| 大规模并发 | 不做压测 | 架构上预留水平扩展能力 |
| 高可用部署 | 单实例 | 无状态设计，支持多实例扩展 |
| 数据持久化 | 可用轻量存储 | 接口抽象，支持替换为生产级存储 |
| 监控告警 | 日志级别 | 预留监控接入点 |
| CI/CD | 手动部署 | 标准化构建脚本，支持接入流水线 |

### 不裁剪项

- **用户交互体验**：完整验证，追求真实体感
- **AI 智能体验**：完整验证，覆盖核心智能场景
- **安全性**：完整验证，不因原型性质降低标准
- **代码质量**：遵循编码规范，保持可读性
