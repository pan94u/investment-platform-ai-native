# 规划基线（Plan Baseline）

> 版本：v0.3 | 日期：2026-03-10 | 状态：共创中
>
> 本文档定义"我们想要构建什么"。交付完成后，通过代码实现反写 `design-baseline.md`（"我们实际构建了什么"），
> 两者交叉比对，得出：✅ 已实现 / ✂️ 已裁剪 / ⏳ 待实现。

---

## 一、产品愿景

用 AI 原生的方式重新定义投资备案体验，证明三件事：

1. **备案比发邮件更简单** — 对话式发起，AI 预填充，零学习成本
2. **审批比看邮件更有价值** — 上下文丰富，风险可见，决策有据
3. **备案数据能产生战略洞察** — 主动发现趋势和异常，辅助管理决策

## 二、用户角色

| 角色 | 说明 | AI 原生体验 |
|------|------|------------|
| 发起人 | 各领域/产业提交备案的人员 | 对话式发起、文档驱动、AI 预填 |
| 审批人 | 直属上级 + 集团审批人 | AI 摘要、风险面板、意见建议 |
| 管理员 | 战略部配置人员 | AI 辅助配置表单和流程 |
| 管理层 | 战略部领导 | 主动洞察、对话式查询、异常推送 |

> 原始需求中的"项目负责人"和"查看角色"在 PoC 中合并到管理层视角验证。

## 三、设计方法：四种流程意图

每个流程环节按真实意图分类，采用不同策略：

| 意图 | 策略 | AI 角色 |
|------|------|--------|
| **底线要求** | 完整保留，自动执行 | 守门员 |
| **留痕** | 保留必要部分，系统自动完成 | 记录员 |
| **相关授权** | 只留核心必要，风险驱动分级 | 风险顾问 |
| **提效** | 全面 AI 原生化 | 执行助手 |

> 详见 `docs/design/ai-native-ux-design-v2.md`

## 四、五层架构与功能规划

```
Layer 4: 底线守护层（规则引擎 + AI）— 贯穿全程
  └─ 数据校验、金额规则、权限边界、AI 安全、数据一致性

Layer 3: 主动洞察层（AI 驱动）— Phase 3
  └─ 底线预警、趋势分析、对话查询、异常推送

Layer 2: 智能辅助层（AI 增强）— Phase 2
  └─ 对话式发起、文档提取、审批摘要、风险评估、分级审批

Layer 1: 基础能力层（传统能力）— Phase 1
  └─ 数据模型、表单引擎、审批流转、Mock 联动、自动留痕

Layer 0: 留痕基座（自动）— 内置
  └─ 对话存档、操作日志、来源标注、上下文快照
```

## 五、技术架构（规划）

### 5.1 架构决策：基于 Synapse AI 构建

不建独立系统，作为 Synapse AI 平台的业务域模块构建。

| 层 | 选型 | 理由 |
|----|------|------|
| 前端 | Synapse Web（Next.js + shadcn/ui） | 复用平台 Chat UI、决策驾驶舱 |
| 后端 | Synapse Server（Hono）+ MCP Server | 复用 Agent 引擎、合规引擎 |
| AI 能力 | Synapse Agent（MiniMax 2.5 + Claude） | 复用 Model Router |
| 数据存储 | File-based JSON（Synapse 标准） | 与平台一致 |
| 部署方式 | Synapse 整体部署 | 统一部署 |

> 详见 `docs/design/synapse-integration-design.md`

### 5.2 投资备案 = Synapse 域模块

```
投资备案域模块组成：
├── MCP Server: investment-filing      # 10 个工具
├── Personas: 3 个角色配置              # YAML
├── Compliance Rules: 规则集            # YAML
├── Skills: 3 个技能                   # SKILL.md
├── Proactive Tasks: 6 个主动任务       # YAML
├── Decision Metrics: 5 个决策指标      # YAML
└── Org Memory: 组织知识初始条目         # JSON
```

### 5.3 架构原则

| 原则 | 说明 |
|------|------|
| 域模块化 | 所有备案功能通过 MCP Server + Skill + Config 实现，不修改 Synapse 核心 |
| 复用优先 | 合规、审批、留痕、主动智能全部复用 Synapse 基础设施 |
| 配置驱动 | 角色、规则、任务通过 YAML 配置，不硬编码 |
| 安全默认 | 继承 Synapse 合规引擎的 Pre-Hook + Post-Hook 双阶段守护 |

### 5.4 Synapse 平台需求清单

| 优先级 | 数量 | 关键项 |
|--------|------|--------|
| P0（阻塞） | 5 | 多步审批流、审批状态机、文件上传、文档提取、表单渲染 |
| P1（提升） | 5 | 待办中心 UI、Persona 动态切换、自定义 Action、事件总线增强、指标数据源 |
| P2（锦上添花） | 4 | Rich Card、飞书审批集成、数据导出、多 Persona 协作 |

> 完整清单见 `docs/design/synapse-integration-design.md` 第四章

## 八、验证度量

### 提效度量

| 度量维度 | 指标 | 目标 |
|---------|------|------|
| 发起效率 | 发起备案步骤数 | 传统 8 步 → AI 3 步（-62%） |
| 审批效率 | 审批处理步骤数 | 传统 5 步 → AI 1~2 步（-60%~-80%） |
| 查询效率 | 数据查询步骤数 | 传统 6 步 → AI 1 步（-83%） |
| 洞察能力 | 异常发现方式 | 被动查看 → 主动推送（质变） |
| AI 准确性 | 表单预填准确率 | ≥ 85% |

### 底线度量

| 度量维度 | 指标 | 目标 |
|---------|------|------|
| 规则拦截 | 不合规提交拦截率 | 100% |
| 数据一致性 | 系统联动成功率 | 100%（失败自动告警） |
| AI 安全性 | 安全测试通过率 | 100% |

### 留痕度量

| 度量维度 | 指标 | 目标 |
|---------|------|------|
| 自动留痕率 | 留痕项中无需人工操作的比例 | ≥ 80% |
| 留痕完整性 | 任意备案可追溯完整决策链 | 100% |

### 授权度量

| 度量维度 | 指标 | 目标 |
|---------|------|------|
| 低风险审批提效 | 低风险备案审批耗时 | 较标准流程缩短 50%+ |
| 高风险决策质量 | 审批人获得的上下文信息量 | 较传统模式提升（定性） |

## 九、基线比对规则

每个 Phase 交付后，执行以下流程：

```
1. 从代码实现反写 design-baseline.md（AI 自动扫描 + 人工确认）
2. 逐条比对 plan-baseline vs design-baseline
3. 标注状态：
   ✅ 已实现 — 功能与规划一致
   ✂️ 已裁剪 — 有意识地裁剪，记录原因
   ⏳ 待实现 — 尚未开始
   🔄 变更   — 实现与规划有差异，记录差异点
4. 比对结果写入 docs/baselines/comparison-phaseN.md
5. 根据比对结果调整后续 Phase 规划
```

> 原型可以丢弃，但规划与实现的偏差必须被记录和理解。
