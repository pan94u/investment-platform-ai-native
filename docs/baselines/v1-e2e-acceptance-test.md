## 场景 8b：同人捏合 — 两级审批人为同一人

> 验证 OrgProvider 的同人捏合逻辑：当 L1 和 L2 审批人是同一个人时，
> 系统自动合并为一级审批，通过即 completed，跳过 L2。
> 这是审批流引擎的 P0 核心能力。

### TC-8b.1 确认同人捏合前提条件

**说明**：当前种子数据中 L1（李四/supervisor）和 L2（王五/group_approver）是不同的人，
正常情况下不会触发同人捏合。需要确认以下任一方式可触发：
- 数据库中将李四同时设为 supervisor 和 group_approver
- 或存在某个领域/部门下同一个人同时覆盖两级审批

**操作**：检查当前系统是否有同人捏合的测试入口（可能需要通过 API 调用验证）

**预期**：
- [ ] 如果前端无法直接触发，记录此场景需 API 级别验证
- [ ] 如果可以触发，继续 TC-8b.2

### TC-8b.2 同人捏合流程（API 辅助验证）

**操作**（如需 API 辅助）：
1. 临时修改数据库，让某个用户同时为 supervisor 和 group_approver
2. 以张三创建并提交备案
3. 观察审批链

**预期**：
- [ ] 提交后只生成 **1 条**审批记录（而非 2 条）
- [ ] 该审批记录 level=1
- [ ] 审批通过后，备案直接变为"已完成"（completed），跳过 pending_level2
- [ ] 审计日志：创建 → 提交 → 一级通过 → 已完成（无二级）

## 关键观察点

1. **状态流转是否正确**：draft → pending_level1 → pending_level2 → completed，以及 rejected/recalled 分支
2. **审批链是否自动创建**：提交后无需手动指定审批人
3. **留痕是否自动完整**：每个操作都有审计日志，无需人工记录
4. **角色隔离是否有效**：发起人不能审批，viewer 不能创建，普通用户不能转办
5. **仪表盘数据是否准确**：统计数字与列表实际条数一致
6. **传统模式步骤数**：记录 8 步基线，供后续 V2 对话式（3 步）对比
