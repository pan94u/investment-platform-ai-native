# 规划基线（Plan Baseline）

> 版本：v0.2 | 日期：2026-03-10 | 状态：共创中
>
> 本文档定义"我们想要构建什么"。交付完成后，通过代码实现反写 `design-baseline.md`（"我们实际构建了什么"），
> 两者交叉比对，得出：✅ 已实现 / ✂️ 已裁剪 / ⏳ 待实现。

---

## 一、产品愿景

用 AI 原生的方式重新定义投资备案体验，证明三件事：

1. **备案比发邮件更简单** — 对话式发起，AI 预填充，零学习成本
2. **审批比看邮件更有价值** — 上下文丰富，风险可见，决策有据
3. **备案数据能产生战略洞察** — 主动发现趋势和异常，辅助管理决策

## 二、用户角色

| 角色 | 说明 | AI 原生体验 |
|------|------|------------|
| 发起人 | 各领域/产业提交备案的人员 | 对话式发起、文档驱动、AI 预填 |
| 审批人 | 直属上级 + 集团审批人 | AI 摘要、风险面板、意见建议 |
| 管理员 | 战略部配置人员 | AI 辅助配置表单和流程 |
| 管理层 | 战略部领导 | 主动洞察、对话式查询、异常推送 |

> 原始需求中的"项目负责人"和"查看角色"在 PoC 中合并到管理层视角验证。

## 三、三层架构与功能规划

```
Layer 3: 主动洞察层（AI 驱动）— Phase 3
  └─ 异常检测、趋势分析、主动推送、对话查询

Layer 2: 智能辅助层（AI 增强）— Phase 2
  └─ 对话式发起、文档提取、审批摘要、风险提示、意见建议

Layer 1: 基础能力层（传统能力）— Phase 1
  └─ 数据模型、表单引擎、审批流转、Mock 联动、数据存储
```

## 四、技术架构（规划）

### 4.1 技术选型

| 层 | 选型 | 理由 |
|----|------|------|
| 前端 | React + Next.js | 生态成熟，AI 组件丰富，SSR 支持 |
| 后端 | Node.js (Express/Fastify) | 快速开发，与 AI API 集成自然 |
| AI 能力 | Claude API | 自然语言理解、文档分析、摘要生成、数据洞察 |
| 数据存储 | PostgreSQL | 结构化数据 + JSON 字段支持动态表单 |
| 部署方式 | Docker Compose（本地） | PoC 单机部署，预留 K8s 扩展 |

> 原始需求指定 Java+MySQL，PoC 选择更轻量的技术栈以加速验证。生产化阶段可根据结论重新选型。

### 4.2 架构原则

| 原则 | 说明 |
|------|------|
| AI 层与业务层分离 | AI 能力通过统一接口调用，业务逻辑不依赖具体 AI 实现 |
| 外围系统抽象 | Mock 与真实接口通过适配器模式切换 |
| 无状态设计 | 服务无状态，会话状态存储在数据库 |
| 接口优先 | 前后端通过 REST API 通信，预留 WebSocket |
| 安全默认 | AI 输出不自动执行，人机边界明确 |

## 五、非功能性需求

| 维度 | 规划要求 | PoC 做法 | 设计预留 |
|------|---------|---------|---------|
| 安全性 | 完整验证 | 不裁剪 | — |
| 交互体验 | 完整验证 | 不裁剪 | — |
| AI 智能体验 | 完整验证 | 不裁剪 | — |
| AI 响应时延 | < 5 秒 | 验证可接受性 | 流式响应 |
| 并发性能 | 设计预留 | 不做压测 | 无状态 + 水平扩展 |
| 高可用 | 设计预留 | 单实例 | 多实例就绪 |
| 可观测性 | 设计预留 | 日志级 | 预留监控接入点 |

## 六、里程碑

| Phase | 目标 | 预计 Session 数 | 验收标准 |
|-------|------|----------------|---------|
| 0 | 项目初始化 + 交付框架 | 1-2 | 框架就绪，首次提交 ✅ |
| 1 | 基础能力层 | 3-5 | 备案 CRUD + 审批流转 + Mock 联动跑通 |
| 2 | 智能辅助层 | 3-5 | 对话式备案 3 步完成、审批摘要+风险面板可用 |
| 3 | 主动洞察层 | 2-4 | 看板可用、对话查询跑通、AI 洞察可生成 |
| 4 | 安全加固 | 1-3 | 6 项安全能力全部验证通过 |
| 5 | 集成验证 + 结论 | 1-2 | 端到端走通、对比度量完成、结论报告输出 |

## 七、验证度量

| 度量维度 | 指标 | 目标 |
|---------|------|------|
| 发起效率 | 发起备案步骤数 | 传统 8 步 → AI 3 步（-62%） |
| 审批效率 | 审批处理步骤数 | 传统 5 步 → AI 2 步（-60%） |
| 查询效率 | 数据查询步骤数 | 传统 6 步 → AI 1 步（-83%） |
| 洞察能力 | 异常发现方式 | 被动查看 → 主动推送（质变） |
| AI 准确性 | 表单预填准确率 | ≥ 85% |
| AI 安全性 | 安全测试通过率 | 100% |

## 八、基线比对规则

每个 Phase 交付后，执行以下流程：

```
1. 从代码实现反写 design-baseline.md（AI 自动扫描 + 人工确认）
2. 逐条比对 plan-baseline vs design-baseline
3. 标注状态：
   ✅ 已实现 — 功能与规划一致
   ✂️ 已裁剪 — 有意识地裁剪，记录原因
   ⏳ 待实现 — 尚未开始
   🔄 变更   — 实现与规划有差异，记录差异点
4. 比对结果写入 docs/baselines/comparison-phaseN.md
5. 根据比对结果调整后续 Phase 规划
```

> 原型可以丢弃，但规划与实现的偏差必须被记录和理解。
