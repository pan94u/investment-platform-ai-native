# AI 原生用户体验设计 v2

> 日期：2026-03-10 | 基于四种流程意图分析升级
>
> 核心方法：**拆解每个流程环节的真实意图，底线保留、留痕精简、授权收窄、提效用 AI。**

---

## 一、流程意图分析框架

任何业务流程中的每个环节，都可以归入四种意图之一：

| 意图 | 定义 | PoC 策略 |
|------|------|---------|
| **底线要求** | 不可违反的业务规则、合规约束 | **完整保留**，用规则引擎 + AI 自动执行 |
| **留痕** | 记录发生了什么，供事后追溯 | **保留必要部分**，AI 对话本身即留痕 |
| **相关授权** | 获取权力许可才能推进 | **只留核心必要**，简单事项免审或轻审 |
| **提效** | 让流程更快、更省力 | **全面 AI 原生化** |

## 二、现有流程环节意图拆解

### 2.1 备案发起环节

| 操作 | 传统设计 | 真实意图 | AI 原生策略 |
|------|---------|---------|------------|
| 选择业务大类 | 手动选菜单 | 提效 | AI 从自然语言自动识别 |
| 选择子场景 | 手动选二级菜单 | 提效 | AI 自动匹配 |
| 选择项目/法人 | 下拉框选择 | 底线（必须关联正确主体） | AI 推荐 + 人确认 |
| 填写备案时间 | 手动填写 | 留痕 | 系统自动记录，无需用户操作 |
| 填写发起人 | 自动带出 | 留痕 | 系统自动记录 |
| 填写业务部门 | 手动选择 | 留痕 | 从用户身份自动带出 |
| 填写涉及金额 | 手动输入 | 底线（金额校验）+ 留痕 | AI 从文档/对话提取，人确认 |
| 填写备案内容 | 手动写多行文本 | 留痕 + 底线（内容完整） | AI 从对话/文档生成，人确认 |
| 上传见证性文档 | 手动上传 | 留痕（证据链） | 保留，但文档内容 AI 自动提取 |
| 保存草稿 | 手动保存 | 提效 | 自动保存，无需手动 |

**结论**：发起环节 10 个操作中，6 个是提效（AI 全自动），2 个是留痕（系统自动记录），2 个是底线（AI 提取 + 人确认）。用户实际需要做的只有：**说清意图 + 确认关键字段 + 上传文件**。

### 2.2 审批环节

| 操作 | 传统设计 | 真实意图 | AI 原生策略 |
|------|---------|---------|------------|
| 发起人直属上级审批 | 固定节点 | 授权 | **重新评估**：见下方分级策略 |
| 集团审批人审批 | 固定节点 | 授权 + 底线 | 核心保留，AI 提供决策上下文 |
| 查看表单详情 | 逐字段阅读 | 底线（了解内容） | AI 摘要替代 |
| 查看附件 | 逐页翻看 | 底线（审核证据） | AI 附件摘要 + 原件可查 |
| 填写审批意见 | 手动填写 | 留痕 | AI 起草，人确认/修改 |
| 驳回/转办 | 手动操作 | 授权 | 保留 |
| 超时催办 | 系统定时 | 提效 | 保留 |

### 2.3 审批授权分级（核心升级）

**传统设计的问题**：所有备案不分轻重，都走"直属上级 → 集团审批人"两级链条。但实际上：
- 一笔 500 万的常规投资备案，和一笔涉及对赌目标大幅下调的备案，风险完全不同
- 一个之前已备案 3 次的成熟项目，和一个首次备案的新项目，所需审批力度不同

**AI 原生策略**：根据风险等级自动调整授权链条。

```
AI 风险评估引擎
    │
    ├─ 低风险（常规备案、金额小、无异常信号）
    │   └─ 轻审批：集团审批人确认即可，直属上级仅抄送
    │
    ├─ 中风险（一般备案、金额中等）
    │   └─ 标准审批：直属上级 → 集团审批人
    │
    └─ 高风险（对赌变更频繁、金额大、异常信号）
        └─ 加强审批：直属上级 → 集团审批人 + 风险提示标红
```

**风险评估维度**：

| 维度 | 低风险信号 | 高风险信号 |
|------|----------|----------|
| 金额 | < 2000 万 | > 5000 万 |
| 对赌变更次数 | 首次 | 12 个月内第 2 次及以上 |
| 对赌变更幅度 | 降幅 < 20% | 降幅 > 30% |
| 项目历史 | 正常运行 | 多次备案、频繁变更 |
| 领域集中度 | 该领域备案频率正常 | 该领域近期备案异常密集 |

> 底线：风险评估结果仅作为审批链调整建议，管理员可配置是否启用。评估逻辑完全透明、可审计。

### 2.4 通知与联动环节

| 操作 | 传统设计 | 真实意图 | AI 原生策略 |
|------|---------|---------|------------|

## 四、留痕体系重构

### 传统留痕 vs AI 原生留痕

| 传统做法 | 问题 | AI 原生做法 |
|---------|------|-----------|
| 用户手动填每个字段 | 用户负担，且可能填错 | AI 提取 + 人确认，确认动作本身即留痕 |
| 审批人手写意见 | 敷衍、模板化 | AI 起草 + 人采纳/修改，保留修改记录 |
| 操作日志记录按钮点击 | 只知道"谁点了同意"，不知道为什么 | 记录完整决策上下文（看了什么、AI 提示了什么） |
| 固定格式归档 | 事后很难理解当时情况 | 对话记录 + 决策上下文一并归档 |

### 自动留痕清单

| 留痕内容 | 留痕方式 | 人工操作 |
|---------|---------|---------|
| 备案发起时间 | 系统自动 | 无 |
| 发起人身份 | 系统自动（SSO） | 无 |
| 备案内容（结构化） | AI 提取 + 人确认 | 确认 |
| 信息来源标注 | 系统标注每个字段的来源（AI 提取/系统带出/人工输入） | 无 |
| 见证性文档 | 用户上传 | 上传 |
| AI 风险评估结果 | 系统自动 | 无 |
| 审批链路与时间线 | 系统自动 | 无 |
| 审批意见 | AI 起草 + 人确认/修改 | 确认或修改 |
| 审批人看到的上下文 | 系统快照（摘要+风险面板） | 无 |
| 对话完整记录 | 系统自动保存 | 无 |

> **关键洞察**：对话记录天然是最完整的留痕。传统表单只记录"最终值"，而对话记录了"意图 → 理解 → 确认"的完整链条，留痕质量反而更高。

## 五、底线保障体系

### 5.1 底线规则分类

| 底线类型 | 规则示例 | 执行方式 |
|---------|---------|---------|
| **数据完整性** | 必填字段不可为空 | 规则引擎自动校验，阻断提交 |
| **业务规则** | 基金投项目金额 < 1 亿 | 规则引擎自动校验，阻断提交 |
| **数据一致性** | 对赌目标变更必须回写战投系统 | 审批通过后自动执行，失败告警 |
| **关联正确性** | 备案必须关联到真实项目/法人 | 从外围系统验证，AI 推荐 + 人确认 |
| **权限边界** | 发起人只能操作有权限的项目 | 数据源按权限过滤 |
| **AI 安全边界** | AI 不自动审批、不越权访问 | 架构层面强制 |

### 5.2 底线检查时机

```
发起阶段                        审批阶段                     完成阶段
    │                              │                          │
    ▼                              ▼                          ▼
 提交前自动校验                  AI 不替代人做决策             系统联动自动执行
 · 必填项完整                    · 摘要和建议仅供参考          · 对赌目标回写
 · 金额规则                      · 人必须做出选择              · 法人结果回传
 · 项目关联有效                  · 决策过程完整留痕            · 失败自动告警+重试
 · 不通过 → 阻断提交             · 风险等级透明可查
```

## 六、授权模型精简

### 传统授权 vs 精简授权

```
传统模式（一刀切）：
  所有备案 → 直属上级审批 → 集团审批人审批

精简模式（风险驱动）：
  ┌─ 低风险 → 集团审批人确认（直属上级仅抄送）
  ├─ 中风险 → 直属上级 → 集团审批人（标准链路）
  └─ 高风险 → 直属上级 → 集团审批人（标红 + 强化上下文）
```

### 授权精简原则

| 原则 | 说明 |
|------|------|
| 风险驱动 | 审批力度匹配风险等级，不搞一刀切 |
| 透明可控 | 风险评估逻辑公开，管理员可调整阈值 |
| 底线不变 | 集团审批人节点始终保留（核心授权） |
| 灵活配置 | 管理员可选择启用分级审批或保持传统模式 |

## 七、四种意图的 AI 能力映射

| 意图 | AI 扮演的角色 | 具体能力 |
|------|-------------|---------|
| **底线要求** | 守门员 | 规则自动校验、数据一致性保障、权限边界执行 |
| **留痕** | 记录员 | 对话自动存档、字段来源标注、决策上下文快照 |
| **相关授权** | 风险顾问 | 风险评估、审批链路建议、上下文提供 |
| **提效** | 执行助手 | 意图识别、表单填充、文档提取、摘要生成、洞察分析 |

## 八、交互模式升级对比

| 操作 | v1 设计 | v2 设计（意图驱动） | 变化 |
|------|--------|-------------------|------|
| 发起备案 | 对话→AI填表→确认 | 对话/上传→AI填表+底线检查+风险预判→确认 | 增加底线自动守护和风险透明 |
| 低风险审批 | 统一审批页面 | 精简卡片，一键确认 | 授权收窄，审批提效 |
| 高风险审批 | 统一审批页面 | 强化上下文+风险标红+意见建议 | 授权加强，决策有据 |
| 留痕 | 用户手动填写留痕 | 系统自动留痕（对话+来源标注+上下文快照） | 留痕更完整，用户更轻松 |
| 通知 | 固定模板 | AI 生成摘要式通知（含风险等级） | 通知即信息，不只是提醒 |

## 九、安全性设计（不裁剪）

与 v1 一致，补充意图层面的安全约束：

| 安全维度 | 所属意图 | 设计方案 |
|---------|---------|---------|
| AI 输入安全 | 底线 | 用户输入清洗后送入 LLM，防 prompt injection |
| AI 输出校验 | 底线 | LLM 输出不直接执行，需用户确认 |
| 数据权限隔离 | 底线 | AI 只访问当前用户有权限的数据 |
| AI 操作审计 | 留痕 | AI 建议了什么、用户采纳了什么，完整记录 |
| 人机边界 | 底线 + 授权 | AI 不自动审批，所有授权决策必须由人做出 |
| 风险评估透明 | 底线 | 评估逻辑可查、可审计、可配置 |
| 敏感数据处理 | 底线 | 脱敏处理 / 私有化部署 |
