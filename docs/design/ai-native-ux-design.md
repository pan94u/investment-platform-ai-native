# AI 原生用户体验设计

> 日期：2026-03-10 | 基于需求价值校验报告重新设计
>
> 设计原则：**对话优先、表单其次；上下文丰富、审批有据；洞察主动、数据说话。**

---

## 一、设计理念

### 从"表单系统"到"AI 助手"

传统系统的核心交互是：**人围着表单转**。
AI 原生系统的核心交互是：**AI 围着人转**。

```
传统模式                              AI 原生模式

选菜单 → 选场景 → 填表单              说一句话 / 传个文件
→ 传附件 → 提交 → 等审批              → AI 理解意图 → 自动填充
                                     → 人确认 → 提交 → 智能审批
```

### 三个设计支柱

| 支柱 | 服务对象 | 核心体验 |
|------|---------|---------|
| **对话式备案** | 发起人 | 比发邮件更简单 |
| **上下文审批** | 审批人 | 比看邮件更有价值 |
| **主动洞察** | 管理层 | 从数据中看到趋势和风险 |

## 二、用户旅程设计

### 旅程 1：对话式备案（发起人）

#### 场景 A：自然语言发起

```
┌─────────────────────────────────────────────────────┐
│  💬 备案助手                                         │
│                                                     │
│  用户: 我要给海川项目做一个对赌目标变更的备案           │
│                                                     │
│  AI: 我找到了「海川项目」的信息：                      │
│      · 项目类型：直投（并购）                          │
│      · 当前对赌目标：2026年净利润不低于5000万           │
│      · 所属领域：智慧住居                              │
│                                                     │
│      请问：                                          │
│      1. 新的对赌目标是什么？                           │
│      2. 变更原因是？                                  │
│      3. 有见证性文档需要上传吗？                       │
│                                                     │
│  用户: 新目标改为2026年净利润不低于3000万，             │
│        因为行业下行，双方协商调整。文件在这里 [上传]     │
│                                                     │
│  AI: 我帮你整理好了备案表单，请确认：                   │
│      ┌──────────────────────────────────┐            │
│      │ 备案类型：直投项目对赌目标变更      │            │
│      │ 项目名称：海川项目                 │            │
│      │ 原对赌目标：2026年净利润≥5000万    │            │
│      │ 新对赌目标：2026年净利润≥3000万    │            │
│      │ 变更原因：行业下行，双方协商调整     │            │
│      │ 见证性文档：协商纪要.pdf ✓         │            │
│      │                                  │            │
│      │    [确认提交]    [修改]            │            │
│      └──────────────────────────────────┘            │
│                                                     │
│  ⚠️ AI 提示：该项目近12个月已有1次对赌变更记录，        │
│     本次为第2次变更，审批人可能会关注变更频率。          │
└─────────────────────────────────────────────────────┘
```

#### 场景 B：文档驱动发起

```
用户上传一份会议纪要 / 协议文件
    │
    ▼
AI 自动提取：
  · 识别备案类型（对赌变更 / 投资 / 退出 / 法人新设）
  · 匹配项目/法人（从战投/法务系统检索）
  · 提取关键信息（金额、目标、原因等）
    │
    ▼
生成预填表单 → 用户确认 → 提交
```

#### 场景 C：快速备案（极简模式）

对于简单场景（如法人信息备案），AI 提供极简路径：

```
用户: 备案一下，智慧住居要新设一个叫"海川科技"的法人

AI: 收到，我来整理：
    · 备案类型：法人新设
    · 法人名称：海川科技
    · 所属领域：智慧住居
    · 需要你补充：注册资本、经营范围、见证性文档

    或者你直接上传法人设立的相关文件，我来提取。
```

#### 设计要点

| 要点 | 说明 |
|------|------|
| 零学习成本 | 不需要知道"直投备案在哪个菜单"，说自然语言即可 |
| AI 预填充 | 项目信息、法人信息、发起人信息全部自动获取 |
| 智能提示 | 提交前给出风险提示、历史关联，帮发起人预判审批关注点 |
| 容错设计 | AI 理解错了可以随时纠正，不需要重新开始 |
| 草稿自动保存 | 对话中途离开，下次回来继续 |

#### 设计要点

| 要点 | 说明 |
|------|------|

| 风险面板 | 自动关联历史数据，呈现异常信号 |
| 同类参考 | 给出基准线，让审批人知道这笔备案在大盘中的位置 |
| 意见建议 | AI 起草审批意见，审批人可采用/修改/自写 |
| 一键操作 | 简单案件一键批过，复杂案件有充分上下文辅助判断 |

#### 设计要点

| 要点 | 说明 |
|------|------|
| 洞察优先于数字 | 不是"本月备案32件"，而是"本月备案同比+60%，主因是法人新设" |
| 异常主动推送 | 不等管理层来看，AI 发现异常主动通知 |
| 对话式探索 | 任何维度的数据查询用自然语言完成 |
| 趋势 > 快照 | 展示变化趋势和模式，而非静态数字 |

## 三、AI 能力矩阵

| AI 能力 | 应用场景 | 技术实现 |
|---------|---------|---------|
| **自然语言理解** | 解析用户发起备案的意图、提取关键信息 | LLM 意图识别 + 实体提取 |
| **文档理解** | 从上传的会议纪要/协议中提取备案信息 | LLM 文档分析 |
| **智能填充** | 根据项目上下文预填表单字段 | 外部系统数据 + LLM 推理 |
| **风险分析** | 识别异常模式（频繁变更、金额异常等） | 规则引擎 + LLM 分析 |
| **摘要生成** | 备案摘要、附件摘要、审批意见建议 | LLM 摘要 |
| **数据洞察** | 趋势分析、异常检测、对比分析 | 数据聚合 + LLM 解读 |
| **自然语言查询** | 管理层用自然语言查询任意备案数据 | Text-to-SQL / RAG |

## 四、交互模式对比

### 传统模式 vs AI 原生模式

| 操作 | 传统模式（步骤数） | AI 原生模式（步骤数） | 体验提升 |
|------|------------------|--------------------|---------|
| 发起对赌变更备案 | 8 步（登录→菜单→场景→选项目→填表→传附件→校验→提交） | 3 步（说意图→确认预填→提交） | **-62%** |
| 审批一笔备案 | 5 步（打开→逐字段阅读→翻附件→写意见→点同意） | 2 步（看摘要+风险→一键审批） | **-60%** |
| 查"去年退出了哪些项目" | 6 步（打开→设筛选条件→选日期→选类型→查询→翻页） | 1 步（直接问） | **-83%** |
| 发现异常趋势 | 被动（需要人主动去看数据并分析） | 主动（AI 发现并推送） | **质变** |

## 五、渐进式体验策略

AI 原生不是"全部推倒重来"，而是在传统能力上叠加 AI 层：

```
Layer 3: 主动洞察层（AI 驱动）
  └─ 异常检测、趋势分析、主动推送、对话查询

Layer 2: 智能辅助层（AI 增强）
  └─ 表单预填、文档提取、审批摘要、风险提示、意见建议

Layer 1: 基础能力层（传统能力）
  └─ 结构化表单、审批流转、系统联动、数据存储、权限控制
```

**PoC 验证策略**：Layer 1 做最小可用，Layer 2 完整验证，Layer 3 重点验证。

## 六、安全性设计（不裁剪）

| 安全维度 | 设计方案 |
|---------|---------|
| AI 输入安全 | 用户输入经过清洗后再送入 LLM，防止 prompt injection |
| AI 输出安全 | LLM 输出经过校验，不直接执行，需用户确认 |
| 数据权限 | AI 只能访问当前用户有权限的数据，不越权推理 |
| 审计追溯 | 所有 AI 辅助操作记录日志（AI 建议了什么、用户采纳了什么） |
| 人机边界 | AI 不自动审批，所有审批决策必须由人做出 |
| 敏感数据 | 金额、对赌目标等敏感字段不发送到外部 AI（使用私有化部署或脱敏处理） |
